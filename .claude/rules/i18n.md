# Internationalization (i18n)

Translation policy for ComboTrack. Non-negotiable for any new or edited UI: **no hardcoded user-facing text** — every string lives in a message catalog and is read through the i18n runtime.

## Concept

`i18n` = **internationalization** (i + 18 letters + n) — the practice of externalizing all user-facing copy so the UI can render in any language. It is the *goal*; `next-intl` is the *tool*. The app ships **French (`fr`) only** for now, but the architecture is built so adding `en` (or others) later requires **no component changes** — only new catalog files.

## Tool — next-intl

`next-intl` is the chosen library: RSC-first i18n for the Next.js App Router, JSON catalogs, typed keys. We use the **single-locale "without i18n routing"** setup — **no `[locale]` URL segment, no middleware**. Upgrading to URL-routed locales later is additive.

## Layout (created in Batch 0 — see `.claude/tasks/21-i18n-french-extraction/`)

- `next.config.ts` — wrapped by `createNextIntlPlugin()`
- `src/i18n/request.ts` — `getRequestConfig` returns a fixed `locale: "fr"` and assembles the messages
- `src/messages/fr/<namespace>.json` — **one file per feature namespace** (landing, video, admin, common…)
- `app/layout.tsx` — `NextIntlClientProvider` + dynamic `lang={locale}` on `<html>`
- `global.d.ts` — `IntlMessages` augmentation for compile-time key autocompletion

## Conventions

- **Never hardcode** user-facing text (JSX, `placeholder`, `aria-label`, `title`, `alt`, toasts, `metadata`, server-action errors). Add a key to the right namespace and read it.
- **Server Components**: `const t = await getTranslations("ns")`. **Client Components** (`"use client"`): `const t = useTranslations("ns")`.
- **Page metadata**: translate via `getTranslations` inside `generateMetadata`.
- **Namespacing**: one namespace per feature domain; keys in `camelCase`, semantic (`video.noteForm.placeholder`). Shared namespaces: `common` (Enregistrer / Annuler / Supprimer / Suivant… + generic validation/errors) and `meta`. **Feature-specific validation and server-action error messages live inside the feature namespace** (`strategyMatrix.validation.*`, `match.errors.*`) — never a global `validation.json` — so parallel batches never edit a shared file.
- **One JSON file per namespace** so parallel extraction batches edit separate files → no merge conflicts.
- **Interpolation** `t("k", { name })`; **plurals** via ICU `{count, plural, =0 {…} one {# …} other {# …}}`. Never string-concatenate or use ternaries for grammar.
- **Zod messages**: schemas store stable **keys** (e.g. `"strategyMatrix.validation.levelLabelRequired"`), translated at render by the shadcn `FormMessage` (`src/components/ui/form.tsx`), adapted once. Keeps schemas pure static modules, safe for both client (react-hook-form) and server (next-safe-action).
- **FGC vocabulary stays in English** even inside French copy (oki, neutral, punish, whiff, anti-air, frame trap, mixup, wakeup, footsies, meter…) — see `design-system.md`. Translate the prose, keep the jargon (e.g. `note-templates.ts` `content` → FR, but `name`/`category` stay EN).
- **Accessibility**: translated `aria-label`s remain mandatory on icon-only controls — see `accessibility.md`.
- Use `??` not `||`. Update `CHANGELOG.md` for every batch.

## Adding a locale later

Add `src/messages/<locale>/*.json`, make `request.ts` resolve the locale dynamically (cookie / header / URL), optionally introduce `[locale]` routing. **No existing `t()` call changes.**
